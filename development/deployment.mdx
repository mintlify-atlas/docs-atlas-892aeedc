## Production Platform

The site is deployed to **[Vercel](https://vercel.com)**, the recommended hosting platform for Next.js applications. Vercel provides zero-configuration deployments, automatic HTTPS, edge caching, and built-in analytics.

**Production URL:** `https://v0-cidteswebsite.vercel.app`

## How Deployments Work

<Steps>
  <Step title="Push to the main branch">
    Every push to the `main` branch triggers an automatic production deployment on Vercel. No manual steps are required.

    ```bash
    git add .
    git commit -m "feat: update certification page content"
    git push origin main
    ```
  </Step>

  <Step title="Vercel runs the build">
    Vercel detects the Next.js project and automatically runs:

    ```bash
    next build
    ```

    This compiles the app, generates static pages, and bundles client-side JavaScript.
  </Step>

  <Step title="Deployment completes">
    Vercel assigns the new build to the production URL and updates the CDN edge nodes. The previous deployment remains available as an immutable preview if you need to roll back.
  </Step>
</Steps>

<Note>
  Pull request branches also receive automatic **preview deployments** with unique URLs. This lets you review changes before merging to `main`.
</Note>

## Build Configuration

| Setting | Value |
|---|---|
| **Build command** | `next build` |
| **Output directory** | `.next` (auto-detected by Vercel) |
| **Node.js version** | 18.x or 20.x (set in Vercel project settings) |
| **Package manager** | pnpm (detected via `pnpm-lock.yaml`) |
| **Framework preset** | Next.js (auto-detected) |

## Next.js Production Settings

The `next.config.mjs` includes settings relevant to production:

```javascript
const nextConfig = {
  eslint: {
    ignoreDuringBuilds: true,  // Build succeeds even if lint errors exist
  },
  typescript: {
    ignoreBuildErrors: true,   // Build succeeds even if type errors exist
  },
  images: {
    unoptimized: true,         // Images served without Next.js optimisation
  },
}
```

<Warning>
  `ignoreBuildErrors: true` and `ignoreDuringBuilds: true` mean TypeScript and ESLint errors will **not** block deployments. It is strongly recommended to fix all type and lint errors locally before pushing, as runtime bugs can still reach production.
</Warning>

## Analytics

`@vercel/analytics` version `1.3.1` is included as a dependency. To activate it, the `<Analytics />` component from `@vercel/analytics/react` must be rendered in `app/layout.tsx`:

```typescript
import { Analytics } from "@vercel/analytics/react"

export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>
        {children}
        <Analytics />
      </body>
    </html>
  )
}
```

Once added, page-view data becomes visible in the **Vercel Analytics** dashboard under your project.

## Manual Production Build

To test the production build locally before deploying:

```bash
pnpm build    # Compile and optimise
pnpm start    # Serve on http://localhost:3000
```

This runs the same `next build` → `next start` pipeline that Vercel uses, so any build-time errors will surface here first.

## Environment Variables

If you add environment variables in the future, configure them in **Vercel → Project → Settings → Environment Variables**. Variables prefixed with `NEXT_PUBLIC_` are exposed to the browser; all others remain server-only.
